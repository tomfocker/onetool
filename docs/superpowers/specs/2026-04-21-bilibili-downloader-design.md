# B 站本地下载工具设计

**日期：** 2026-04-21

**状态：** 已确认，等待实现计划

## 目标

在 `onetool` 内新增一个本地 B 站下载工具，支持用户在工具内扫码登录、粘贴视频或剧集链接、选择具体分 `P` 或剧集条目，并将当前账号本来就有权限正常播放的非 `DRM` 内容下载到本地。

本轮目标聚焦四件事：

- 提供独立于浏览器 Cookie 的工具内扫码登录能力。
- 支持对普通视频、多分 `P` 视频、番剧/影视剧集链接做元数据解析。
- 支持用户选择具体条目、清晰度和导出模式后执行下载。
- 支持原始流下载与本地 `FFmpeg` 合并导出 `MP4`。

## 用户意图与约束

本轮确认后的要求如下：

- 形态是并入 `onetool` 的本地工具，不做浏览器扩展。
- 登录态来源为工具内扫码登录，不读取本机浏览器 Cookie。
- 第一版只支持用户粘贴链接后下载，不做站内搜索。
- 导出模式需要支持“仅流下载”和“合并为 `MP4`”两类路径。
- 对多分 `P` 视频或番剧页，先展示可选条目，再由用户明确选择目标。
- 只支持当前账号可正常播放、非 `DRM` 的内容，不做绕过权限、破解或解密。

据此，本轮不做“外部下载器套壳”或“嵌入浏览器扩展逻辑复用”，而是按 `onetool` 现有的 `service + ipc + tool` 架构提供原生工具能力。

## 问题定义

要把这项能力稳定放进 `onetool`，需要先解决四个核心问题：

### 1. 登录态来源必须可控

如果直接读取外部浏览器 Cookie，会带来来源不清、失效不可控和跨环境兼容问题。该工具需要独立维护自己的登录态，避免和用户浏览器环境耦合。

### 2. 链接语义不稳定

用户粘贴进来的链接可能是：

- 普通 `BV` 视频链接
- 多分 `P` 视频链接
- 番剧或影视剧集链接
- 带当前分 `P`、剧集、查询参数的页面链接

因此需要先把链接标准化，再把“页面解析”和“实际下载”拆开。

### 3. 媒体资源形态不统一

可下载资源可能是：

- 分离的 `DASH` 音视频流
- 单文件媒体流
- 存在多个清晰度、编码和音频轨的组合

工具必须先对播放信息做归一化，再生成适合用户选择的导出选项。

### 4. 下载与合并必须可观测

这不是浏览器下载链接跳转场景。工具需要自己管理请求、文件写入、临时目录、进度、取消和 `FFmpeg` 合并，否则无法给出稳定的桌面端体验。

## 方案比较

### 方案 A：Electron 本地工具 + 主进程下载服务

特点：

- 使用 `onetool` 现有 Electron 主进程能力承载登录、解析、下载和合并。
- 渲染层只负责展示状态、表单、列表和进度。
- 通过独立 service 保存登录态和任务状态。

优点：

- 最贴合当前仓库结构。
- 登录、下载、文件写入和 `FFmpeg` 调用都可控。
- 不依赖外部浏览器环境。

问题：

- 需要维护一套接口封装、状态持久化和错误处理逻辑。

推荐采用此方案。

### 方案 B：内嵌浏览器页，依赖网页上下文完成解析

特点：

- 在工具内部打开网页或嵌入登录页，再借助页面上下文拿播放数据。

优点：

- 部分页面数据获取路径表面上更直接。

问题：

- 工具会演化成半个浏览器，明显偏离 `onetool` 定位。
- 页面结构与脚本环境变动风险大。
- 状态边界不清晰，后续维护成本更高。

不采用此方案。

### 方案 C：封装外部下载器

特点：

- 调用已有第三方下载程序，把 `onetool` 当壳层。

优点：

- 初期开发快。

问题：

- UI、登录、条目选择、进度和错误反馈都不受控。
- 与当前仓库的工具一体化程度低。
- 后续扩展和维护会明显受限。

不采用此方案。

## 设计结论

第一版采用“Electron 本地工具页 + 主进程下载服务 + 独立登录态存储 + 本地 `FFmpeg` 合并”的方案。

核心目标不是一次做成完整下载管理器，而是先把以下链路做稳定：

- 能登录
- 能解析
- 能选条目
- 能下载
- 能合并
- 出错时知道错在哪

## 工具范围

第一版支持以下能力：

- 工具内扫码登录
- 粘贴视频或剧集链接
- 解析元数据和可选条目
- 选择剧集或分 `P`
- 选择清晰度与导出模式
- 下载当前单个目标条目
- 原始流导出与本地 `MP4` 合并

第一版明确不做：

- 站内搜索
- 批量整季下载
- 多任务队列管理
- 断点续传
- `DRM` 内容处理
- 权限绕过或解密

## 交互设计

### 登录流

- 工具页首次进入时显示未登录状态和“开始扫码登录”主按钮。
- 用户点击后，主进程请求一次登录二维码数据，并返回二维码展示信息、轮询键和过期时间。
- 渲染层展示二维码并轮询登录状态。
- 登录成功后，主进程持久化最小必要会话信息，并回传用户昵称、头像和会话有效状态。
- 工具页支持“退出登录”和“重新登录”。

### 链接解析流

- 用户粘贴链接后先执行标准化解析，提取 `bvid`、`aid`、`ep_id`、`season_id`、当前 `p` 等参数。
- 工具先请求元数据，只生成标题、封面、类型、分 `P` 列表、剧集列表和可访问状态。
- 如果是多分 `P` 或剧集页面，用户必须先显式选择目标条目。
- 条目选定后，再请求该条目的播放信息与导出选项。

### 下载执行流

- 用户选择导出模式、清晰度和目标条目后，再开始下载。
- 下载状态按阶段显示：`解析中`、`下载视频流`、`下载音频流`、`合并中`、`完成`、`失败`。
- 下载过程中允许取消当前任务。
- 完成后展示输出路径和最近一次任务结果。

## 导出模式设计

第一版提供以下导出模式：

- `仅视频流`
- `仅音频流`
- `原始流分别下载`
- `合并为 MP4`

处理原则：

- 若播放信息为 `DASH`，优先下载独立音视频流。
- 若用户选择 `合并为 MP4`，调用本地 `FFmpeg` 做无转码合并，优先 `-c copy`。
- 若只有单文件流，则允许直接保存，不强制拆成音视频。
- 若合并失败，保留已下载的原始流文件用于排障与手动处理。

## 架构设计

### 主进程职责

主进程负责：

- 登录二维码申请与登录状态轮询
- 登录态持久化与会话校验
- 链接标准化解析
- 视频、分 `P`、剧集元数据请求
- 播放信息归一化
- 媒体流下载、进度汇报和取消
- 临时文件管理与 `FFmpeg` 合并

## 数据模型

至少需要以下运行态模型：

- `loginSession`
  - 用户基础信息
  - 会话有效期
  - 登录态摘要
- `parsedLink`
  - 链接类型
  - 标题
  - 封面
  - 目标列表
  - 默认选中项
- `streamOptions`
  - 清晰度列表
  - 编码信息
  - 音频轨信息
  - 当前模式是否可导出
- `downloadTask`
  - 任务状态
  - 当前阶段
  - 进度
  - 输出目录
  - 结果文件路径
  - 错误信息

共享类型建议放入 `src/shared`，避免主进程和渲染层维护两套不一致的字段定义。

## 存储与文件路径

### 登录态存储

- 不读取浏览器 Cookie。
- 在 `app.getPath('userData')` 下保存该工具独立使用的会话数据。
- 只保存最小必要字段，如 `SESSDATA`、`bili_jct`、用户基础信息和过期时间。

### 文件输出

- 最终输出目录由用户选择或使用默认下载目录。
- 临时文件放在 `userData/cache/bilibili-downloader`。
- 任务成功后清理临时文件。
- 合并失败时保留最近一次原始流文件和错误信息。

## 错误处理

工具必须明确区分以下失败类型：

- 未登录或登录态失效
- 链接格式不支持
- 链接有效但条目不可访问
- 当前账号无播放权限
- 当前清晰度或导出模式不可用
- 下载流失败
- `FFmpeg` 合并失败

错误处理原则：

- 不把所有错误折叠成一个“下载失败”。
- 用户可选择项在渲染前就尽量禁用，而不是点击后才报错。
- 合并失败时不丢原始下载结果。

## 安全与边界

实现边界必须清晰：

- 仅处理当前登录账号已能正常访问的非 `DRM` 内容。
- 不读取外部浏览器 Cookie。
- 不做权限绕过、解密、破解或区域限制规避。
- 如果接口返回权限不足、会员限制或不可访问状态，只做清晰提示并停止流程。

## 测试策略

至少补以下测试：

- `BilibiliDownloaderService`
  - 链接标准化解析正确区分普通视频、分 `P` 视频和剧集链接。
  - 元数据归一化能生成稳定的条目列表。
  - `DASH` 与单流资源能生成正确导出选项。
  - 下载任务状态流转正确。
  - 合并失败时保留原始流文件。
- `bilibiliDownloaderIpc`
  - 关键 IPC 参数校验与错误路径正常。
- 渲染层工具页
  - 未登录、登录中、已登录状态切换正确。
  - 多分 `P` / 剧集选择后才能进入下载动作。
  - 不可用清晰度和导出模式会被禁用。

对远端接口本身不做脆弱的在线单元测试，优先围绕归一化逻辑、状态机和本地文件处理补测试。

## 非目标

- 不做断点续传。
- 不新增浏览器扩展侧代码。
- 不引入需要额外原生安装步骤的新运行时。

## 风险与回退

主要风险：

- 登录接口或页面字段变化导致扫码流程失效。
- 不同链接类型的元数据结构差异导致解析分支复杂。
- `DASH` 资源组合过多，首版导出选项归一化不完整。
- 合并链路出错时，如果清理过早会丢失排障证据。

回退原则：

- 如果首版流选项归一化不稳定，优先保留“单条目 + 基本清晰度 + 合并 `MP4`”主链路。
- 如果某类链接解析不稳定，允许临时只支持已验证的链接类型，不影响整体工具落地。
- 不允许为了追求更多站点行为兼容而引入浏览器扩展式注入方案。
