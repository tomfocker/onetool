# 任务栏美化设计

**日期：** 2026-04-21

**状态：** 已确认，等待实现计划

## 目标

新增一个独立的“任务栏美化”工具页，让用户在 `onetool` 内手动切换 Windows 任务栏样式，并把“毛玻璃”效果作为第一版的核心体验重点。

这个功能的核心不是完整复刻 `TranslucentTB` 的托盘应用和动态规则系统，而是在现有 Electron 工具箱架构里提供一个稳定、可恢复、可持久化的手动任务栏样式控制器。

## 用户意图与约束

本轮确认后的要求如下：

- 第一版只做手动切换样式，不做动态规则联动。
- 优先把毛玻璃效果做好，透明和亚克力作为补充预设。
- 工具首版可以明确限定为仅 Windows 可用，并优先支持 Windows 11。
- 其他平台或不支持的 Windows 版本不做兼容实现，只显示不可用提示。
- 当前阶段以“先把功能做出来”为优先目标，不以商业化许可边界约束实现路线。

据此，第一版默认按“用户已接受 Windows-only、手动模式优先”的模式推进，不再额外引入多平台方案或动态策略设计。

## 方案比较

### 方案 A：独立原生小助手

特点：

- 任务栏控制能力独立为单独 exe 或 dll。
- Electron 主程序只负责 UI、配置和控制指令。
- 后续替换底层实现或拆分分发边界更容易。

问题：

- 首版开发成本更高。
- 需要额外设计进程管理、IPC 协议、打包与生命周期管理。
- 对“先做出手动切换效果”的目标来说，工程投入偏大。

### 方案 B：主进程内置任务栏服务

特点：

- 直接复用现有 `service + ipc + preload + tool page + settings` 结构。
- 首版只做手动预设切换时，链路短、验证快。
- 打包和启动恢复逻辑都能直接放进当前主进程体系。

问题：

- Windows API 逻辑会先落在主进程侧。
- 后续如果要抽成独立原生小助手，需要再做一次边界迁移。

推荐采用此方案。

原因：

- 最符合当前仓库结构和首版目标。
- 能最快形成“工具页 -> IPC -> 主进程服务 -> 任务栏生效”的可验证闭环。
- 通过把 Windows API 封装在适配层中，后续仍然保留抽离为独立原生模块的余地。

## 第一版范围

### 覆盖能力

- 新增独立“任务栏美化”工具页。
- 支持四个手动预设：
  - `系统默认`
  - `透明`
  - `毛玻璃`
  - `亚克力`
- 支持调节强度，首要服务 `毛玻璃` 效果。
- 支持调节色调蒙层，改善图标和文字可读性。
- 支持一键恢复系统默认样式。
- 持久化最近一次成功应用的配置，并在应用重启后尝试恢复。
- 展示当前系统支持状态和降级原因。

### 页面结构

页面结构分为五块：

1. 顶部状态卡
2. 效果预设区
3. 强度与色调调节区
4. 应用与恢复操作区
5. 系统兼容性提示区

### 交互行为

- 用户切换预设后立即应用效果。
- 用户调整强度和色调后立即应用效果。
- 当前配置只有在成功应用后才会写入持久化设置。
- 启动恢复时如果能力检测失败或应用失败，自动回退到系统默认并提示原因。

## 非目标

- 不做“有窗口时切换样式”“最大化时切换样式”等动态规则。
- 不做开始菜单、搜索、任务视图、任务栏线条等联动逻辑。
- 不做系统托盘菜单控制。
- 不做多平台兼容实现。
- 不做复杂的自定义规则 DSL 或导入导出配置能力。

## 预设定义

第一版的预设需要统一抽象为工具内部的样式配置，而不是把 UI 直接绑定到底层 Windows API 参数。

建议的预设语义如下：

- `系统默认`
  - 撤销 `onetool` 施加的任务栏外观修改。
- `透明`
  - 以较轻的透明层为主，不强调模糊。
  - 目标是获得清透感，同时保持任务栏内容可读。
- `毛玻璃`
  - 作为默认推荐项。
  - 以稳定、柔和、可读性高为优先目标。
  - 强度滑杆主要调节这一模式的模糊或材质强度。
- `亚克力`
  - 作为进阶选项。
  - 仅在当前系统版本与能力检测允许时启用。
  - 若底层能力不足，应明确降级或置灰，而不是悄悄失败。

## 能力检测与降级策略

Windows 不同版本上，任务栏材质支持和表现存在差异。第一版必须在应用效果前做能力检测，而不是让用户在失败后自行猜测问题。

能力检测至少覆盖：

- 当前平台是否为 Windows。
- 当前 Windows 主版本与构建号。
- 当前预设在本机上是否允许启用。
- 当前任务栏窗口是否可被找到和修改。

降级策略如下：

- 非 Windows：工具页显示不可用，不提供操作。
- Windows 但不支持当前材质：对应预设置灰，并显示原因。
- 启动恢复失败：自动恢复系统默认，不保留失败状态。
- 运行时应用失败：保留当前已生效状态，并向页面返回错误信息。

## 架构设计

### 总体分层

实现按三层拆分：

- `Renderer`
  - 只负责展示状态、发送用户意图、展示错误与兼容性提示。
- `TaskbarAppearanceService`
  - 负责预设语义、配置持久化、能力判断、启动恢复、失败回退。
- `WindowsTaskbarAdapter`
  - 只负责寻找任务栏窗口并调用 Windows API 修改样式。

这个边界是第一版的关键。即使当前仍然选择主进程内置实现，也必须把 Windows 专属代码收敛在适配层中，避免整个主进程到处散落平台特化逻辑。

### 文件结构

建议新增或修改以下文件：

- 新增 `src/main/services/TaskbarAppearanceService.ts`
- 新增 `src/main/services/windows/WindowsTaskbarAdapter.ts`
- 新增 `src/main/ipc/taskbarAppearanceIpc.ts`
- 修改 `src/main/index.ts`
- 修改 `src/preload/createElectronBridge.ts`
- 新增 `src/renderer/src/tools/TaskbarAppearanceTool.tsx`
- 修改 `src/renderer/src/data/tools.ts`
- 修改 `src/shared/types.ts`
- 修改 `src/main/services/SettingsService.ts`

### 服务职责

`TaskbarAppearanceService` 至少负责：

- 读取和输出任务栏配置状态。
- 把 UI 预设映射为底层样式参数。
- 做平台与版本能力检测。
- 应用预设、恢复默认。
- 在应用启动时恢复上次成功配置。
- 对失败情况执行保护性回退。

`WindowsTaskbarAdapter` 至少负责：

- 查找主任务栏窗口。
- 如有必要，查找多显示器扩展任务栏窗口。
- 调用 Windows 能力设置透明、模糊或材质效果。
- 撤销已施加的外观修改。
- 返回底层失败原因，供服务层转换为用户可理解的错误。

## 数据模型

### 配置模型

第一版建议在 `AppSettings` 中新增轻量任务栏配置字段，例如：

- `taskbarAppearancePreset`
- `taskbarAppearanceIntensity`
- `taskbarAppearanceTint`
- `taskbarAppearanceEnabled`

说明：

- 首版任务栏配置规模较小，不需要像下载整理那样单独扩展 `GlobalStore`。
- 只有“成功应用过”的配置才应被持久化。

### 运行态模型

运行态需要和持久化配置区分开，至少包含：

- 当前平台是否支持该工具。
- 当前系统版本信息。
- 每个预设是否可用。
- 当前已生效预设。
- 当前错误信息。
- 最近一次恢复是否成功。

## IPC 设计

新增独立 IPC：

- `taskbar-appearance-get-status`
- `taskbar-appearance-apply-preset`
- `taskbar-appearance-restore-default`

如首版需要即时调节色调或强度，也可以统一由 `apply-preset` 承载完整配置对象，而不是拆成多个高频 IPC。

返回数据至少要包含：

- 当前支持状态
- 当前预设
- 每个预设的可用性
- 当前平台和版本提示
- 最近一次错误

## 渲染层设计

### 新增工具页

新增 `src/renderer/src/tools/TaskbarAppearanceTool.tsx`。

页面职责：

- 读取主进程返回的能力状态与当前配置。
- 展示预设卡片、强度滑杆、色调调节器。
- 在用户操作时调用 IPC 应用效果。
- 在失败时清晰展示错误，不自行猜测降级结果。

### 页面反馈

页面必须明确区分三种状态：

- `不可用`
  - 当前平台或系统版本不支持。
- `可用但部分预设受限`
  - 工具可用，但某些预设需要置灰并显示原因。
- `可用`
  - 所有首版预设都可以操作。

### 文案方向

首版页面文案应强调：

- `毛玻璃` 是推荐效果。
- `亚克力` 受系统版本影响，可能受限。
- 任何时候都可以一键恢复到系统默认。

## 错误处理策略

任务栏相关功能如果失败，不能让用户陷入“到底当前是什么状态”的不确定状态。

第一版要求：

- 应用失败时返回明确错误，不写入新设置。
- 恢复默认失败时也要返回错误，但仍需尽力撤销本次修改。
- 启动恢复失败时自动清理为系统默认，并记录失败信息。
- 页面层不缓存乐观状态，以主进程返回的已生效状态为准。

## 测试策略

### 共享逻辑测试

至少覆盖：

- 预设到底层配置的映射。
- 不同 Windows 版本下的能力判断。
- 持久化配置与运行态状态合并逻辑。
- 应用失败时不污染持久化设置。

### 主进程服务测试

至少覆盖：

- `getStatus` 在非 Windows 和 Windows 下的行为差异。
- `applyPreset` 在支持和不支持预设时的行为。
- `restoreDefault` 的回退逻辑。
- 启动恢复只恢复最近一次成功配置。
- 适配层返回错误时，服务层能正确转换为稳定状态。

### 手工验证

由于任务栏材质效果依赖真实 Windows 环境，第一版必须补充手工验证：

- Windows 11 上切换 `系统默认 / 透明 / 毛玻璃 / 亚克力`
- 调整强度与色调后的视觉变化
- 恢复默认后的任务栏状态
- 关闭并重新打开 `onetool` 后的恢复行为

## 验证标准

满足以下条件即可认为第一版达到可用：

- 工具页在 Windows 11 上可以打开并展示当前系统支持状态。
- `系统默认`、`透明`、`毛玻璃` 至少能稳定切换。
- `亚克力` 在支持时可启用，在不支持时明确置灰并说明原因。
- `毛玻璃` 的图标与文字可读性保持在可接受范围。
- 切换失败不会把任务栏卡在不可恢复的异常状态。
- 一键恢复能把任务栏回退到系统默认。
- 成功应用过的配置在应用重启后能恢复。
- 测试、类型检查和构建通过。
