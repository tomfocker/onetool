# Windows NTFS 极速空间扫描设计

**日期：** 2026-04-20

**状态：** 已确认，待评审

## 目标

把当前“空间清理”从普通递归扫描升级为一套真正接近成熟磁盘分析器体验的极速模式。

本轮目标不是继续优化现有 Node.js 递归遍历，而是新增一条 **Windows 专项、NTFS 专项** 的扫描路径，使本地 NTFS 盘在整盘扫描时能做到：

- 首屏结果快速出现
- UI 不冻结
- 顶层目录占用很快可见
- 后续细节持续流式补全

这套能力只承诺覆盖：

- Windows
- 本地 NTFS 卷

不承诺覆盖：

- exFAT
- FAT32
- 网络盘
- WSL 挂载路径
- 任意普通子目录路径的极速扫描

## 用户意图与边界

用户要求很明确：

- 整盘扫描不能再是现在这种普通递归慢扫体验
- 如果做不到成熟软件的效果，这个功能宁可不用
- 可以接受极速模式只支持 Windows 的 NTFS 本地盘

因此本轮设计不做折中版本，不以“并发更多一点的 Node 遍历”冒充极速模式。

## 结论

### 推荐方案

采用：

- **Rust 独立扫描器**
- **直接读取 NTFS MFT**
- **Electron 主进程只负责桥接与回退**

这是当前项目里最现实、最可维护、最接近成熟软件效果的路线。

### 不采用的路线

#### 1. 继续优化 Node 递归扫描

即使增加并发、缓存、分阶段 UI，也只能改善体感，不能达到成熟工具的整盘极速扫描效果。

#### 2. Electron 原生 addon（N-API）

能做，但维护成本高于独立进程方案：

- Electron ABI 兼容更麻烦
- 崩溃隔离更差
- 调试和发布链更重

### 为什么选独立扫描器

- 主进程崩溃隔离更好
- 调用边界清晰，Electron 仍保持 TypeScript 主导
- 以后可以单独替换或升级扫描器
- 打包时可以作为 `extraResources` 一起分发，和 `ffmpeg.exe` 的处理方式一致

## 功能范围

### 支持的极速模式目标

极速模式第一版只支持：

- 本地盘根路径，例如 `C:\`、`D:\`
- 且卷格式必须是 `NTFS`

### 明确不支持极速模式的对象

- `exFAT`
- `FAT32`
- `ReFS`
- 映射网络盘
- UNC 路径
- WSL / Linux 挂载路径
- 非根路径目录，例如 `D:\Work`

### 不支持时的行为

产品行为需要明确、不能含糊：

- 对支持的卷：显示 `极速扫描`
- 对不支持的卷：明确提示 `当前路径不支持极速扫描`

第一版推荐行为：

- **盘符根目录但不是 NTFS**：禁用极速扫描，可选择普通扫描
- **普通目录路径**：只允许普通扫描
- **网络盘/异常卷**：直接禁用极速扫描并说明原因

不允许：

- 用户以为自己在用极速模式，实际上偷偷退回慢扫而不说明

## 交互设计

### 工具页变化

“空间清理”页增加扫描模式层：

- `极速扫描（NTFS）`
- `普通扫描`

实际交互建议：

- 当用户选择根盘符且检测到为 NTFS 时，默认高亮 `极速扫描`
- 当用户选择普通目录时，自动切到 `普通扫描`
- 页面上明确显示当前模式

### 极速扫描的用户感知

极速模式不能等所有数据齐了才显示页面。

要求：

- 很快给出卷级基本信息
- 很快给出顶层目录占用
- 后台继续补充：
  - 更深层目录
  - 最大文件列表
  - 完整目录树

### 页面状态

新增或明确以下文案：

- `极速扫描准备中`
- `极速扫描中（NTFS）`
- `当前卷不支持极速扫描`
- `已回退到普通扫描`

## 架构设计

### 总体结构

```text
Renderer
  -> preload bridge
  -> Main process SpaceCleanupService
  -> NtfsFastScannerBridge
  -> bundled rust scanner executable
  -> NTFS volume / MFT
```

### 角色划分

#### Renderer

只负责：

- 发起扫描请求
- 消费流式会话更新
- 展示当前模式、进度、结果

不负责：

- 判断卷格式
- 解析扫描器输出
- 管理子进程生命周期

#### Main process

负责：

- 判断当前路径是否满足极速模式条件
- 选择 `极速扫描` 或 `普通扫描`
- 启动、取消、回收原生扫描器进程
- 把扫描器流式输出转换为现有 `SpaceCleanupSession`
- 在不支持时返回明确原因

#### Rust 扫描器

负责：

- 验证目标卷是否为本地 NTFS 卷
- 直接读取 NTFS MFT
- 构建文件记录索引
- 聚合目录大小
- 输出增量结果

## 扫描原理

### 普通扫描为什么慢

当前 `SpaceCleanupService` 基于：

- `readdir`
- `stat`
- 递归遍历

它必须真的逐层走进去读每个目录，这在整盘扫描下天然慢。

### 极速模式为什么快

极速模式不走普通文件系统递归，而是直接读：

- `NTFS Master File Table (MFT)`

MFT 已经是 NTFS 卷上的文件记录总表。扫描器通过读取 MFT，可以更快拿到：

- 文件记录
- 父子关系
- 文件大小
- 目录关系

然后在内存里聚合目录大小，而不是靠普通目录遍历一层层相加。

### 第一版输出策略

扫描器输出分阶段：

1. `volume-info`
   - 卷类型
   - 卷标
   - 是否支持极速模式
2. `top-level-summary`
   - 顶层目录体积分布
3. `largest-files`
   - 当前已知大文件排行
4. `tree-snapshot`
   - 更完整的目录树快照
5. `complete`
   - 最终结果

这样前端可以先显示“足够有用”的结果，而不是等全量结束。

## 扫描器集成方式

### 形式

新增一个独立 Windows 可执行文件，例如：

- `resources/space-scan/ntfs-fast-scan.exe`

打包时通过 `electron-builder` 的 `extraResources` 分发。

### 主进程桥接层

新增一个主进程桥接模块，例如：

- `NtfsFastScannerBridge`

职责：

- 拼装命令行参数
- 启动子进程
- 读取 stdout 流
- 把 stdout 逐条解析为结构化事件
- 统一收集 stderr 和退出码
- 处理取消

### 输出格式

推荐扫描器使用：

- **JSON Lines**

每一行一条 JSON 事件。

原因：

- 易于流式处理
- 易测试
- 出错边界清晰
- 以后扩展字段不会破坏兼容

示意事件：

```json
{"type":"volume-info","mode":"ntfs-fast","rootPath":"D:\\","filesystem":"NTFS"}
{"type":"top-level-summary","directories":[...],"filesScanned":123456}
{"type":"largest-files","items":[...]}
{"type":"complete","summary":{...},"tree":{...}}
```

## 现有服务改造

### SpaceCleanupService

`SpaceCleanupService` 不再只拥有一条扫描路径，而是负责模式编排：

- 判断目标路径
- 选择扫描实现
- 维护统一 `SpaceCleanupSession`

建议内部拆成：

- `SpaceCleanupService`
- `NtfsFastScannerBridge`
- `DirectoryScanner`（保留现有普通扫描逻辑）

### 扫描模式判定

建议增加：

- `mode: 'ntfs-fast' | 'filesystem'`

进入 `startScan()` 时先做判定：

1. 是否 Windows
2. 是否盘符根路径
3. 是否本地卷
4. 文件系统是否 NTFS
5. 极速扫描器是否可用

只要有一项不满足，就不能进入极速模式。

### 会话模型扩展

建议 `SpaceCleanupSession` 增加：

- `scanMode`
- `scanModeReason`
- `isPartial`

说明：

- `scanMode`: 当前实际使用的扫描模式
- `scanModeReason`: 为什么启用或禁用极速模式
- `isPartial`: 当前结果是否仍在增量补全中

## 数据模型

### 扫描器中间数据

Rust 扫描器内部至少需要维护：

- 文件记录索引
- 目录记录索引
- 父子映射
- 路径解析缓存
- 前 N 大文件堆

### 输出给 Electron 的树

仍然尽量复用当前 `SpaceCleanupNode` 结构，这样渲染层不用推翻重写。

但要允许增量场景：

- 顶层目录先返回
- 深层 children 后补

因此需要允许：

- 子节点暂不完整
- 节点附带“是否已完整展开”的元信息

建议新增：

- `isHydrated?: boolean`

## 错误处理

### 必须明确区分的错误

#### 不支持极速模式

这是正常产品分支，不是异常。

例如：

- 非 NTFS
- 非本地卷
- 非根路径

前端应显示正常提示，而不是报错 toast。

#### 极速扫描器不可用

例如：

- 扫描器文件不存在
- 启动失败
- 权限异常

建议：

- 给出明确错误
- 允许用户手动回退普通扫描

#### 扫描器异常退出

例如：

- 退出码非 0
- 输出解析失败

建议：

- 保留已收到的部分结果
- 会话标记 `failed`
- 同时附带 stderr 最后若干行，便于诊断

## 性能目标

第一版不写虚假承诺，但要给出明确目标：

- 对本地 NTFS 盘整盘扫描，首屏顶层结构应在几秒内出现
- UI 全程不冻结
- 顶层目录分布应显著快于现有普通扫描
- 最终全量结果时间允许因盘大小、文件数而变化

这里要强调：

- “成熟软件秒出”的核心是首屏可用结果很快，不一定是全盘最终完全结束也在 1 秒内

## 安全与权限

极速模式只读 NTFS 元数据，不做写操作。

但实现上需要注意：

- 只开放给本地卷读取
- 任何删除动作仍然走现有 Electron 层，不下沉到扫描器
- 扫描器不负责执行文件删除

- NTFS 极速模式判定正确

- JSON Lines 输出正确

### 集成验证

必须在真实 Windows 环境验证：

- NTFS 本地盘极速扫描可启动

- UI 不冻结
- 取消扫描有效

## 验收标准

本轮完成标准：

- `空间清理` 支持 `极速扫描（NTFS）`
- 仅本地 NTFS 根盘可启用
- 首屏目录占用结果显著快于普通递归扫描
- UI 不冻结
- 可取消
- 非支持路径明确提示原因
- 构建、测试、打包链可处理扫描器资源

## 非目标

本轮不做：

- 非 Windows 平台极速扫描
- exFAT/FAT32/ReFS 极速扫描
- 普通子目录极速扫描
- 重复文件检测
- 实时监听磁盘变化
- 多卷并行极速扫描
