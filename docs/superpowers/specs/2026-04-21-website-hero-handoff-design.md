# OneTool 官网首屏 Typography 与 Handoff 改版设计

**日期：** 2026-04-21

**状态：** 已确认，待评审

## 目标

在现有 `website/` 发布页结构不推翻重来的前提下，对首屏进行一次定向改版，解决两个已经明确暴露的问题：

1. 当前主标题排版过于竖向堆叠，字面冲击力有了，但稳定感和高级感不足。
2. 当前首屏碎裂动效虽然有拆解感，但没有真正实现与下方模块的共享元素联动，且滚动过程中存在明显卡顿。

这次改版的目标不是重新设计整页信息架构，而是在现有官网骨架上，把首屏收敛成一个更像正式产品发布页的入口，同时把滚动联动从“示意飞行”升级为“真实落位”。

## 用户意图与确认结论

基于本轮确认，用户希望：

- 左侧主标题继续保留明显的发布页冲击力，但不要再使用现在这种过于拥挤、纵向压缩过重的堆叠方式。
- 标题气质参考更克制、更稳定的发布页排法，而不是海报式竖向大字。
- 首屏碎裂动效不是简单散开或飞向大概方向，而是要让每个功能模块随着滚动真正落到下方对应分页上。
- 联动方式采用共享元素式 handoff，也就是首屏卡片与下方模块在视觉上像同一个元素跨区迁移，而不是两套元素交替出现。
- 整体动效必须比现在更顺，不接受继续靠高频滚动重计算换视觉效果。

## 当前问题

### 1. 标题问题

当前 Hero 主标题虽然足够醒目，但行长、断行和中英混排关系不稳定，导致实际效果更像“被强行竖着塞进去的大字”，而不是成熟发布页里的高密度排版。

具体表现：

- 中文大字纵向堆叠过重。
- `Windows` 一词在版面里权重过高，压住了整句节奏。
- 主标题块整体过高，导致正文与按钮被挤到更靠下的位置。
- 视觉上更像字在撑画面，而不是文字与右侧主视觉共同构图。

### 2. 动效问题

当前 Hero flight card 的 motion 已经具备“breakout / travel / settle”阶段，但仍有两个根本缺陷：

- 目标点计算只是近似朝下方模块移动，没有建立共享元素式的精确落位关系。
- 滚动时对布局信息的读取和同步节奏仍偏重，视觉结果就是滚动过程发虚、发飘、跟手性差。

用户感受到的问题不是“没有动画”，而是动画没有完成故事闭环：它拆开了，却没有真正落下去。

## 改版方向

这次改版采用“苹果式发布页标题 + 共享元素 handoff”方向。

### 标题方向

- 标题保持一句直接的人话表达，不回到抽象 campaign 文案。
- 排版重心从“纵向堆字”改为“横向节奏明确的发布页标题”。
- 中文主句与 `Windows` 的关系从强插入式大块混排，调整为更自然的节奏锚点。
- 力量感主要来自字重、行宽控制、留白与对齐关系，而不是单纯把字做得巨大。

### 动效方向

- 首屏卡片仍然先经历破碎和分层展开。
- 但从中段开始，每张卡片不再朝一个大概方向飞，而是逐步对齐到真实目标模块。
- 在最终阶段，卡片的旋转、缩放、圆角、透明度和高亮状态一起收敛，让用户感觉它已经“成为”下方模块本身。
- 下方模块不是被动出现，而是承担最终接管。

## 非目标

- 不重做整页信息架构。
- 不引入新前端框架。
- 不引入物理引擎、Canvas、WebGL 或复杂 3D runtime。
- 不把联动做成夸张的粒子碎裂或特效秀。
- 不为了 handoff 效果牺牲移动端可读性。

## 设计原则

### 1. 标题必须像正式发布页，而不是海报字墙

标题仍然可以大，但要服从行宽、断行、字重分配和中英混排节奏。用户第一眼应该感受到“高级且稳”，而不是“字很大”。

### 2. Handoff 必须是元素迁移，不是区域高亮

这次联动的成立标准不是“看起来大概飞过去了”，而是用户能明确看到某张 Hero 卡在滚动中变成下方模块的一部分。

### 3. 共享元素的错觉必须优先于绝对精度

视觉上要建立“同一个元素跨区迁移”的感受，但实现方式仍然可以是轻量的 CSS transform + JS 定位，而不是昂贵的真实布局动画系统。

### 4. 性能优先于花哨

如果某个视觉细节会带来明显卡顿，就应当先砍掉它，再保留 handoff 主线。真正重要的是节奏顺、目标准、叙事清楚。

## 视觉设计

### 标题排版方案

主标题改为更横向、更稳定的两段式发布页结构，推荐文案保持现有语义，但调整为如下节奏：

```text
一个应用，收齐
Windows 日常高频工具
```

排版要求：

- 第一行更短，承担节奏起手。
- 第二行更长，承担价值完整表达。
- `Windows` 保留英文，但不再单独撑成超大块。
- 主标题整体不超过三行。
- 标题块最大宽度应明显大于当前版本，避免继续形成竖向塔状结构。

视觉规则：

- 中文与英文统一使用 sans-serif，不引入 serif。
- 主标题字重保持强，但略低于当前最黑最厚的压迫感。
- `Windows` 的字号可以与中文同级或略小一级，不应再抢走整句重心。
- 行距需要略放开，让字与字之间有呼吸，而不是压扁。
- 左侧正文与按钮整体上移，为右侧 handoff 区域释放更多纵向空间。

## Handoff 动效设计

### 目标模块映射

首屏中参与迁移的 Hero 卡片与下方模块建立一一映射：

- `capture` -> 截图 / 录屏场景卡
- `organize` -> 批量重命名 / 文件整理场景卡
- `clipboard` -> 剪贴板或其所属整理模块
- `utility` -> 二维码 / 轻工具模块
- `matrix` -> 工具矩阵主分组

其中 `clipboard` 可以与 `organize` 共用目标区域，但在最终停靠点上应有偏移区分，避免两张卡完全重叠。

### 分阶段节奏

Hero sticky 滚动继续采用分阶段 motion，但阶段语义需要改写为：

1. `rest`
   用户初见完整组合体。
2. `break`
   首屏卡片从核心层拆开，建立破碎感。
3. `travel`
   卡片沿真实目标路径开始下落，不再只是局部散开。
4. `morph`
   卡片接近目标模块时，旋转逐步归零，缩放、圆角和高光同步收敛。
5. `dock`
   Hero 卡片透明消退，下方真实模块接管，形成共享元素式完成态。

建议阶段区间：

- `0.00 - 0.18`: `rest`
- `0.18 - 0.42`: `break`
- `0.42 - 0.78`: `travel`
- `0.78 - 0.94`: `morph`
- `0.94 - 1.00`: `dock`

### 视觉收敛规则

当 Hero 卡片接近目标模块时，需要同步发生以下变化：

- `rotateX / rotateY / rotateZ` 逐渐归零
- `scale` 收敛到接近目标模块尺寸
- `border-radius` 向下方模块的圆角收敛
- 阴影从悬浮态减弱到贴合态
- `opacity` 从 Hero 主体逐步交给下方真实模块
- 下方目标模块同步增加高亮、边框增强和轻微抬升

最终结果不是“飞过去然后消失”，而是“它落进去以后，下面那块就成了它本身”。

## 实现设计

### 结构层

改版仍限定在 `website/` 静态站内完成，涉及：

- `website/index.html`
- `website/style.css`
- `website/script.js`
- `website/hero-motion.js`
- `website/hero-motion.test.cjs`

其中：

- `index.html` 负责标题结构微调，以及为目标模块补充更稳定的绑定标识。
- `style.css` 负责标题视觉重构、共享元素阶段样式和 dock 态接管样式。
- `script.js` 负责缓存目标位置信息、滚动同步调度和 CSS 变量写入。
- `hero-motion.js` 负责阶段计算函数。
- `hero-motion.test.cjs` 负责验证新的阶段边界与 handoff 节奏。

## 数据流

### 1. 目标框采集

页面初始化时读取所有参与映射的目标模块矩形信息，并将其转换为相对于 Hero flight 容器的目标位移。

只在以下时机重算：

- 首次加载
- 字体加载完成后
- `resize`
- `pageshow`

不在每一帧滚动里重算目标框。

### 2. 滚动同步

滚动事件不直接执行完整样式同步，而是只更新一个待处理标记，由 `requestAnimationFrame` 统一读取最新滚动位置并写入 CSS 变量。

这样可以避免：

- 高频 scroll 触发下的重复布局读取
- 同一帧内多次写样式
- 滚动过程的抖动和掉帧

### 3. 阶段计算

`hero-motion.js` 输出的状态需要新增或明确以下字段：

- `break`
- `travel`
- `morph`
- `dock`
- 各目标模块的 `highlight`

阶段计算必须是单向、连续、无回跳的。用户缓慢滚动时，卡片应持续朝目标收敛，不出现短促回弹或目标漂移。

## 性能边界

为确保这次改版不会再次出现明显卡顿，实现必须遵守以下约束：

- 滚动动画只使用 `transform`、`opacity`、必要时少量 `filter`。
- 不在 scroll handler 中调用多次 `getBoundingClientRect()`。
- 不为每张卡单独挂复杂 observer。
- 减少大面积模糊和多层重阴影的同时运动。
- 当 `prefers-reduced-motion: reduce` 生效时，关闭复杂迁移，仅保留静态对齐和简单 reveal。

## 测试与验证

### 自动化测试

需要扩展 `website/hero-motion.test.cjs`，覆盖以下行为：

- 标题与 handoff 阶段仍然在初始进度下静止。
- 中段滚动时卡片已经进入 `travel`，而不是只停留在 `break`。
- 接近末段时 `morph` 已经启用，目标模块高亮开始接管。
- 最末段时 `dock` 状态生效，工具矩阵接管出现在场景卡之后。
- `prefers-reduced-motion` 下不再输出复杂阶段运动值。

### 视觉验证

最终页面至少要满足：

- 主标题第一眼更稳、更像正式发布页，不再像竖排字墙。
- 用户滚动时能明确看到每张 Hero 卡朝真实模块下落。
- 下方模块在视觉上接住 Hero 卡，而不是独立出现。
- 滚动手感明显顺于当前版本，没有持续性抖动。
- 桌面端 handoff 成立，移动端自动降级为更克制的排法与动效。

## 风险与取舍

### 1. 精确落位依赖真实目标框

如果后续继续频繁调整下方模块结构或间距，映射目标就需要同步重算和微调。为降低维护成本，目标绑定必须保持语义稳定。

### 2. 共享元素错觉依赖节奏而不是数学完美

即使目标框是精确的，如果阶段区间、缩放收敛和高亮接管节奏不对，视觉上仍会像两套元素。因此动画调参与结构绑定同等重要。

### 3. 标题克制后会降低“字很大”的直接冲击

这是主动取舍。目标是换来更高的质感和更稳定的构图，而不是继续用超大纵向字墙抢视觉。

## 结论

这次改版应把当前 Hero 从“排版过猛但不稳、动效有拆无落”的状态，推进到“标题更像正式发布页、卡片真正落到对应模块”的状态。

标题部分的关键是把力量感从纵向堆字转成横向节奏与中英混排控制；动效部分的关键是把近似飞行改成共享元素式 handoff，并以性能优先的调度方式保证滚动手感。
