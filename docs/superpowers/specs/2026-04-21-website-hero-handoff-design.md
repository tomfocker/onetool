# OneTool 官网 Hero Handoff 三段式改版设计

**日期：** 2026-04-21

**状态：** 已确认，待评审

## 目标

把当前官网从多段式信息架构收敛成更直接的三段式结构：

1. `首页 Hero`
2. `工具展示`
3. `下载收口`

这次改版要同时解决两个已经确认的问题：

- 当前 handoff 同时指向“日常场景”和“工具矩阵”两层目标，用户看不清卡片到底落到哪里。
- 页面中段过多，导致滚动叙事被拆散，Hero 卡片的下落路径不够直接。

目标不是做新的视觉风格，而是把现有官网骨架压缩成更清晰的发布页叙事：首屏展示零散能力，第二屏接住并整理为工具分组，第三屏完成下载转化。

## 用户确认结论

本轮已确认的决策如下：

- 删除 `日常场景` 页面。
- 删除 `系统支持` 页面。
- 官网只保留 `首页 + 工具展示 + 下载` 三段式结构。
- 第二页不是纯空白过渡，而是保留一小段页头说明。
- 第二页的视觉提示采用轻提示方案，不增加新的“接收区”或中转舞台。
- Hero flight 卡的目标统一改为第二页工具分组，不再同时落向多个 section。

## 当前问题

### 1. 信息架构问题

当前页面路径是：

`Hero -> 日常场景 -> 工具矩阵 -> 系统支持 -> 下载`

问题在于：

- `日常场景` 和 `工具矩阵` 都像接收页，目标语义重复。
- Hero 卡片既指向场景卡，又继续收敛到工具矩阵，造成双层目标。
- 用户滚动到第二页时能看到卡片还在动，但不容易判断最后会落在哪一层。
- `系统支持` 本身信息量不足，却占掉一个独立 section，拉长了页面。

### 2. Handoff 叙事问题

当前 handoff 虽然已经能“往下走”，但叙事仍不够单义：

- 目标分散，用户需要猜测落点。
- 第二页不是最终接收页，导致“看到一半又继续往下”。
- 卡片在结构上没有一个唯一、稳定、可阅读的完成态。

## 改版方向

这次改版采用“真三段式发布页 + 单一目标 handoff”方向。

核心变化只有两件事：

- 页面结构从五段收敛为三段。
- Hero 卡片只落到第二页工具分组，不再跨两层目标。

这样改完后，页面叙事会被压缩成一条清楚的主线：

`Hero 展示能力碎片 -> 工具展示页接住并归类 -> 下载页收口`

## 非目标

- 不重新设计整套视觉语言。
- 不引入新前端框架。
- 不保留“场景页缩编版”作为第二页的隐藏替代品。
- 不新增独立的 handoff 舞台、轨道区或中转层。
- 不在这次设计里扩展新的功能模块或下载逻辑。

## 设计原则

### 1. 页面路径必须单义

用户滚动时，不应该再思考“卡片是落到场景页还是工具矩阵”。第二页就是唯一接收页。

### 2. 第二页先是内容页，再是动效承接页

第二页的本质是工具展示，不是动画舞台。动效只负责帮助理解，不应喧宾夺主。

### 3. 删除就是删除，不做伪保留

既然已经决定删掉 `日常场景` 和 `系统支持`，就不再保留它们的导航、文案残影或结构影子。

### 4. Handoff 必须可读

比起更复杂的 motion，更重要的是用户能一眼看懂卡片正在往哪个工具分组收敛。

## 页面结构设计

### 第一段：首页 Hero

保留内容：

- 顶部导航
- 品牌标识
- 主标题
- 简短说明
- 主 CTA / 次 CTA
- 5 张 Hero flight 卡

职责：

- 用标题和主视觉建立产品理解。
- 把零散工具能力以 flight 卡形式展示出来。
- 为第二页的落位制造起点。

### 第二段：工具展示

保留内容：

- 一小段页头说明
- 3 个工具分组卡

不保留内容：

- 场景页文案
- 场景 01/02/03 卡片
- 系统支持图标或说明条

第二页承担两个职责：

- 文案上告诉用户这里就是工具分组总览。
- 视觉上承担 Hero 卡片的唯一接收页。

推荐页头说明语气：

- 简短、直接。
- 强调“这些常用动作已经被整理成分组”。
- 不再讲使用场景，不再重复系统支持信息。

### 第三段：下载收口

继续保留下载页作为最后一段，不承担 handoff 目标，只承担转化与收口。

## 第二页视觉设计

第二页采用“轻提示”方案，而不是增加新的接收带或中转层。

组成方式：

- 页头一小段说明
- 下方直接是 3 个工具分组卡

当 handoff 发生时，只通过以下方式表达目标：

- 对应分组卡抬升
- 对应分组卡边框变亮
- 对应分组卡背景亮度轻微提升
- 对应分组卡阴影增强

不增加：

- 独立的接收轨道
- 中转占位区
- 单独的“停机坪”视觉层

原因很明确：第二页必须像真实内容页，而不是一层额外动画设备。

## Handoff 设计

### 目标规则

Hero flight 卡统一只落到第二页的 3 个工具分组。

映射关系定为：

- `capture + matrix -> 捕获与处理`
- `organize + clipboard -> 文件与文本`
- `utility -> 更多小工具`

这意味着：

- 不再存在 `#scenarios` 相关目标。
- 不再存在“先落场景，再继续落工具矩阵”的双阶段结构。
- 第二页工具分组就是 handoff 的最终完成态。

### 节奏规则

Hero 卡片的节奏改为：

1. `rest`
   初始展示完整 Hero 组合。
2. `break`
   轻量拆开，建立碎片感。
3. `travel`
   当第二页标题区接近视口后，卡片才开始明显下行。
4. `morph`
   当第二页 3 个分组卡进入主视区后，卡片开始缩放、转正、收敛。
5. `dock`
   对应分组卡抬升接管，Hero 卡贴近并淡出。

关键限制：

- 卡片不能在用户还停留在首屏时就接近完成态。
- 第二页刚露头时，用户必须还能看见卡片正在持续下落。
- dock 完成点必须发生在第二页工具分组真正可读的时候，而不是更早。

### 可读性规则

为了让用户一眼看懂落点，第二页必须满足：

- 3 个工具分组卡的位置稳定，不能在 handoff 过程中再大幅重排。
- 每组的视觉差异足够明确，避免卡片像落到一个模糊的大白块上。
- 接管高亮要和卡片到达同步，而不是提前闪。

## 实现边界

本次改版仍限定在 `website/` 静态站内完成，主要涉及：

- `website/index.html`
- `website/style.css`
- `website/script.js`
- `website/hero-motion.js`
- `website/homepage-structure.test.cjs`
- `website/hero-motion.test.cjs`
- `website/hero-style-contract.test.cjs`

结构调整方向：

- 从 `index.html` 中移除 `日常场景` section。
- 从 `index.html` 中移除 `系统支持` section。
- 更新导航锚点，只保留 `首页 / 工具展示 / 下载`。
- 保留 Hero flight 容器在首屏，不再迁移到第二页。
- 重写第二页工具分组的目标标识，使其成为唯一 dock target。

动效调整方向：

- 移除所有对场景页目标的映射和阶段依赖。
- 将 handoff 的 travel / morph / dock 统一绑定到第二页工具展示区。
- 继续采用 `requestAnimationFrame + 缓存目标框`，避免把布局测量放回滚动热路径。

## 测试与验证

### 自动化测试

需要调整测试覆盖以下行为：

- 页面结构只保留 3 段主区块。
- 导航锚点只剩 `首页 / 工具展示 / 下载`。
- 不再存在 `#scenarios` 和 `#system-support` 的结构断言。
- Hero 卡目标映射只指向第二页 3 个工具分组。
- 中段滚动时卡片仍在 travel，不会在第二页露头前过早完成。
- 末段滚动时第二页工具分组完成接管。
- `prefers-reduced-motion` 下退化为更克制的 reveal。

### 视觉验证

最终页面至少要满足：

- 用户能明确知道 Hero 卡最终落到第二页工具分组。
- 页面中段不再让人感觉“还有一层过渡页”。
- 第二页在没有动效时也成立，仍像一个完整工具展示页。
- 下载页仍然作为第三段自然收口。

## 风险与取舍

### 1. 删除中段内容会让页面更短

这是主动取舍。页面会更短，但换来更强的路径清晰度。

### 2. 第二页信息密度会上升

因为场景解释和系统支持都被删掉，第二页必须自己承担“工具归类说明”。这就是为什么要保留一小段页头说明。

### 3. 分组数量必须克制

如果第二页再继续扩成很多组，唯一落点的可读性又会被稀释。因此工具分组数量应控制在一眼能扫完的范围内。

## 结论

这次改版的本质不是继续调一版动画，而是先把页面语义理顺：

- 首页负责展示能力碎片。
- 第二页负责接住并整理成工具分组。
- 第三页负责下载转化。

只要结构先变成这条单义路径，Hero handoff 才会真正成立，用户也不会再追问“这些卡片到底落到哪里去了”。
