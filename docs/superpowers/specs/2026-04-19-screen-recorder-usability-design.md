# 屏幕录制可用性升级设计

**日期：** 2026-04-19

**状态：** 已确认，待评审

## 目标

在不重写现有录制引擎的前提下，把屏幕录制从“功能可用”提升到“面板驱动下顺手可用”。

本次聚焦两个体验问题：

- 区域录制选择不顺手，选错后重来成本高。
- 录制中的状态反馈不清楚，主界面、悬浮提示、停止流程割裂。

## 用户意图与约束

基于本轮确认，目标体验如下：

- 用户偏好通过面板完成录制准备，而不是热键直达录制。
- 录制模式重点优化 `区域录制` 和 `录制中状态`。
- 区域录制在框选完成后，需要回到面板进行确认。
- 面板内需要支持选区微调，而不是只能“重选”。
- 开始录制后，主界面不应继续原样停留，而应收成一个小型悬浮控制条。

## 非目标

- 不改造成热键直达录制工作流。
- 不新增 WebM 真正编码支持，本次仅修正文案与可选项一致性。
- 不实现复杂的可视化锚点拖拽编辑，先使用数值微调。
- 不重做 FFmpeg 录制链路。
- 不扩展到窗口录制、多轨音频、摄像头叠加等新能力。

## 当前问题

### 1. 区域录制确认链路太弱

当前区域录制只支持框选后回传矩形，面板里仅显示尺寸文案。用户无法确认录到的实际内容，也无法在开始前做小幅修正。

### 2. 录制中状态被分散到多个地方

录制启动后，主窗口被最小化，同时存在一个只显示时间的悬浮提示。用户可见的“录制中”状态与可操作入口割裂，停止和恢复查看主面板都不够顺手。

### 3. 渲染层状态偏零散

当前录屏页的 React 状态以字段拼装为主，缺少清晰的流程阶段定义。随着“选区确认”和“悬浮控制条”加入，继续在现有结构上堆叠会让行为越来越难预测。

### 4. 文案与能力不一致

界面文案提到 `WebM`，但现有实现并未真正支持该格式。这会直接损害用户对录屏功能的信任感。

## 设计方案

采用“面板分阶段流程”方案，在一个工具页内完成录制准备、确认、开始、收起与恢复，不新增独立的大型录屏子系统。

## 交互流程

录屏页改成三段式流程：

1. `录制目标`
2. `录制确认`
3. `录制中控制`

### 阶段 1：录制目标

用户先选择：

- `全屏录制`
- `区域录制`

交互要求：

- 全屏录制时，展示可选屏幕卡片。
- 区域录制时，主操作按钮文案改为 `框选区域`。
- 点击 `框选区域` 后进入选区模式。
- 选区完成后返回面板，不直接开始录制。

### 阶段 2：录制确认

这是本次升级的核心。

当区域录制完成框选后，面板进入 `ready-to-record` 状态，并展示一张完整的选区确认卡片，包含：

- 选区真实预览图
- 左上角坐标 `x / y`
- 尺寸 `width / height`
- `重选区域`
- 数值微调控件

微调策略：

- 支持调整 `x / y / width / height`
- 默认步进为 `1`
- 按住 `Shift` 时步进放大为 `10`
- 每次微调后刷新预览图，保证看到的就是最终录到的内容

全屏录制也进入统一的确认态，但不显示选区预览卡，重点展示：

- 当前屏幕
- 输出格式
- 帧率
- 画质
- 保存位置

### 阶段 3：录制中控制

点击 `开始录制` 后：

- 主录屏页收起，不继续完整停留在前台
- 保留区域边框窗口作为空间提示
- 显示小型悬浮控制条作为唯一操作入口

悬浮控制条至少包含：

- `正在录制`
- 已录制时长
- 当前模式（全屏或区域）
- `停止`
- `返回主面板`

`返回主面板` 只恢复查看与状态同步，不允许在录制中修改关键参数。

## 状态模型

录屏会话应显式定义为以下五个状态：

- `idle`
- `selecting-area`
- `ready-to-record`
- `recording`
- `finishing`

状态约束：

- `idle` 时允许编辑所有草稿配置，不允许停止录制。
- `selecting-area` 时主面板进入等待选区回传状态。
- `ready-to-record` 时允许确认选区与进行数值微调。
- `recording` 时锁定格式、FPS、画质、目标区域等关键参数。
- `finishing` 时禁止重复触发停止，直到 FFmpeg 退出并回传结果。

## 结构设计

### 主进程职责

`ScreenRecorderService` 继续作为录屏唯一真相源，负责：

- 当前录制状态
- 当前会话配置
- 当前选区数据
- 当前预览图
- FFmpeg 录制进程
- 悬浮控制条窗口
- 区域边框窗口

建议把状态聚合成单一会话对象，而不是继续扩散在多个零散字段中。建议字段包括：

- `status`
- `config`
- `selectionBounds`
- `selectionPreviewDataUrl`
- `startedAt`
- `outputPath`

### 渲染层职责

`useScreenRecorder` 从“字段集合”调整为“流程驱动状态”。

渲染层重点维护：

- `draftConfig`
- `selectionDraft`
- `sessionStatus`
- `canStart`
- `isPanelCollapsed`

原则：

- 渲染层负责配置与确认
- 主进程负责录制生命周期
- 两边通过 IPC 同步，而不是各自推断录制实际状态

### 选区预览获取

现有选区能力已经能回传 `bounds`，但不足以支撑确认体验。

本次不增加新的采集实现，直接复用现有 `ScreenshotService.capture(bounds)`：

1. 选区窗口回传 `bounds`
2. 主进程立即抓取该区域截图
3. 将 `bounds + preview` 一并返回给录屏页

这样录屏页中的“选区确认卡”展示的是真实截图，而不是纯尺寸信息。

### 悬浮控制条

当前 `indicatorWindow` 升级为可交互的 mini controller，而不是新增第三类控制窗口。

职责：

- 展示录制状态
- 承担停止入口
- 触发主面板恢复显示

保留 `borderWindow` 作为纯视觉边界提示，不承担操作职责。

## IPC 调整

现有能力基础足够，不需要推翻重做，但需要补齐面向流程的事件。

建议增加或统一为以下事件：

- `selection-confirmed`
  返回 `bounds + preview`
- `recording-session-updated`
  返回 `status / elapsed / mode`
- `recording-panel-expand-requested`
  由悬浮控制条触发，要求主窗口恢复显示

目标是减少渲染层对碎片化事件的依赖，让录屏状态切换更稳定。

## 边界行为

### 区域选择

- 小于最小尺寸的选区直接判无效，不进入确认态。
- 跨屏选区本次不支持，限定在框选所在显示器范围内。
- 选择 `重选区域` 时，必须清掉旧的预览图与旧坐标。

### 微调规则

- 只允许正整数。
- 宽高存在最小值下限，建议为 `64 x 64`。
- 微调越过显示器边界时自动钳制，不弹复杂报错。
- 微调后重新抓取预览，保证预览与实际录制一致。

### 录制中保护

- 进入 `recording` 后锁定所有关键录制参数。
- 点击停止后立即进入 `finishing`，只显示“正在停止...”。
- 如果 FFmpeg 异常退出，主窗口恢复并保留最近一次录制草稿，便于重试。

## 完成与失败反馈

### 成功

成功结束录制时，需要同时发生以下反馈：

- 悬浮控制条销毁
- 主窗口恢复到录屏页
- 明确的成功提示，包含输出路径

### 失败

失败反馈需要分型，而不是统一报“录制失败”：

- `启动失败`
- `录制中断`
- `停止失败`
- `输出异常`

主窗口恢复后，需要保留上次配置和选区数据，降低再次尝试的成本。

## UI 调整范围

### 保留

- 现有录屏页作为统一入口
- 全屏/区域两种模式
- 保存位置、格式、FPS、画质、热键设置

### 调整

- 录制页布局改成更明显的阶段式流程
- 区域录制新增“选区确认卡”
- 开始录制后改为小悬浮控制条
- 录制建议与文案收敛，移除未实现能力的暗示

### 文案修正

必须修正以下不一致：

- 标题副文案中移除 `WebM`
- 建议区中移除 `WebM` 描述
- 格式选项与实际编码能力保持一致，只保留当前真实支持项

## 验收标准

满足以下条件即视为本次设计目标达成：

1. 用户完成一次区域框选后，可以看到真实预览，而不是只有尺寸文字。
2. 用户可以在面板内微调 `x / y / width / height` 后再开始录制。
3. 开始录制后，主页面会收成小悬浮控制条，而不是继续完整留在前台。
4. 点击停止后不会重复停止，也不会误报“异常中断”。
5. 录制完成或失败后，主窗口都能恢复到录屏页，并保留可复用配置。
6. 页面文案与实际录制能力一致，不再出现“支持 WebM”这类误导信息。

## 实施建议

建议按以下顺序实现：

1. 先梳理主进程会话状态与 IPC
2. 再补 `bounds + preview` 确认链路
3. 然后改录屏页阶段式交互与微调控件
4. 最后把现有指示条升级为悬浮控制条，并完成状态收口

这样可以优先把“选区确认”和“录制中控制”两个核心体验稳定下来，减少一次性大改带来的回归风险。
